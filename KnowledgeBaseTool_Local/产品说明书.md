# K-Matrix 助手产品说明书

文档版本：v1.0  
检查日期：2026-05-29  
适用项目：KnowledgeBaseTool_Local  
运行形态：本地 Flask 后端 + 静态前端 + SQLite 系统库 + 本地 PostgreSQL 主业务库

## 1. 产品概述与核心价值

K-Matrix 助手是一套面向企业知识库运营、质量评估、机型适配和治理闭环的本地化管理工具。系统把知识库主数据、机型矩阵、多媒体资源、评分治理、智能映射、修改记录和归档管理放在同一个工作台中，帮助运营人员完成“导入、编辑、校验、同步、评分、治理、归档”的完整流程。

核心价值如下：

1. 统一维护知识库 V1 与 V1T-1：V1 是当前主库，V1T-1 是前刻备份库。系统支持主库编辑、导入、导出、软删除、状态流转和一键同步到前刻库。
2. 将“问答内容”和“适用机型”拆解为矩阵：机型矩阵可以快速查看每条知识覆盖哪些型号，并支持单格、批量、复制、克隆和提交日志。
3. 多媒体资源自动聚合：从知识库图片、视频、文件和跳转链接字段中抽取 URL，形成可预览、可打标、可追溯的资源表。
4. 大模型辅助质量评估与内容优化：支持配置 OpenAI 兼容接口，对问答质量评分，也可对问题、答案、相似问和问题类型做 AI 优化。
5. 治理与审计闭环：修改记录、矩阵提交日志、月度召回治理、智能映射和归档批次共同形成可追踪的知识库治理流程。
6. 本地优先，兼容 Supabase 风格接口：当前配置优先使用本地 PostgreSQL，部分模块仍保留远端 Supabase 或本地 SQLite fallback 能力。

## 2. 目标用户与应用场景

### 2.1 目标用户

1. 知识库运营人员：负责新增、编辑、删除、导入、导出、查重和修订知识。
2. 产品/机型运营人员：负责维护产品型号库、产品分类、机型分类映射和机型矩阵。
3. 质量治理人员：负责评分抽样、批量评分、月度召回治理和结果归档。
4. 内容审核人员：查看修改记录、智能合并导出、提交日志和归档批次。
5. 系统管理员/开发维护人员：负责本地数据库、配置文件、AI Key、Prompt、依赖和迁移脚本维护。

### 2.2 典型应用场景

1. 日常知识库维护：运营人员在“知识库管理”中搜索问题、编辑答案、调整适用型号，系统记录修改日志并把状态置为修订中。
2. 批量导入新版本：通过 Excel/CSV 导入，选择增量更新、批量新增或全量覆盖；系统做字段映射、型号校验、JSON 字段清洗和状态重置。
3. 发布前备份与下游刷新：将 V1 同步到 V1T-1，然后一键同步矩阵、多媒体预览和评分快照。
4. 机型适配调整：在矩阵视图中筛选问题，批量把某机型设为适用/不适用，再提交差异，形成操作日志和修改记录。
5. 质量抽检：从评分模块随机抽样，调用 LLM 评分并导出质量报告。
6. 月度治理：导入每月召回数据，与知识库和评分表关联，筛选低质量、高召回或主库不存在的条目。
7. FAQ 智能映射：导入 FAQ 表和知识库表，按问题/答案相似度比对，人工确认新增或更新建议，并写入修改记录。

## 3. 系统组成与数据边界

### 3.1 前端

前端位于 `link_viewer/`，主文件为 `index.html`、`app_v8.js`、`styles.css` 和 `extra_styles.css`。页面是单页应用，登录后通过左侧导航切换业务模块。前端使用 `API_BASE = /api` 与 Flask 后端通信。

前端辅助机制：

1. 登录失败、接口 401 时显示登录面板。
2. GET 请求在 `api()` 封装中有一次短延迟重试。
3. 知识库分页有 5 分钟缓存和下一页预加载，失败时静默处理。
4. 多媒体表格列宽等部分 UI 配置保存在浏览器 localStorage。
5. 编辑弹窗包含关闭确认、Markdown 工具栏、AI 优化按钮、快捷录入按钮和标签选择入口。

### 3.2 后端

后端入口为 `server.py`，基于 Flask，默认监听 `0.0.0.0:8085`。主要职责包括：

1. 用户登录与 session 管理。
2. 静态前端托管。
3. 知识库、矩阵、链接、评分、治理、归档、智能映射等 API。
4. 本地 PostgreSQL 客户端封装，模拟 Supabase/PostgREST 风格调用。
5. SQLite 系统库管理，包括用户、部分矩阵本地表、提交日志、归档、智能映射任务、运维库 fallback 和 outbox。

### 3.3 数据存储

1. SQLite 系统库：`instance/data.db`。保存用户、部分本地矩阵表、矩阵提交操作、按钮提交日志、归档批次、归档记录、运维库 fallback、智能映射任务和 Supabase outbox。
2. 本地 PostgreSQL 主业务库：由 `supabase_config_local.json` 的 `local_db` 配置连接。核心表包括 `knowledge_base_v1`、`knowledge_base_v1_t1`、`kb_scores`、`kb_recall`、`link_previews`、`knowledge_base_modifications`、`kb_tags`、`kb_item_tags` 等。
3. 配置文件：型号库、机型分类映射、AI 配置、评分配置、主库配置均以 JSON 文件形式保存，其中部分文件在项目内为符号链接，实际指向外层配置目录。
4. Prompt 目录：AI 优化 Prompt 会保存到 `prompt/`，并生成历史版本。

### 3.4 登录与权限边界

系统目前只有登录态控制，没有角色级权限。除 `/`、`/login`、`/api/status` 和 `/api/proxy_image` 外，主要业务 API 均通过 `@login_required` 保护。

注意：`/api/proxy_image` 当前没有登录保护，且会代理外部 URL。此处存在安全风险，详见第 9 章。

## 4. 全量功能模块详解

## 4.1 登录与会话

### 功能范围

1. 打开根路径 `/` 返回前端页面。
2. `/api/status` 返回当前是否已登录。
3. `/login` 校验用户名和密码，成功后写入 Flask-Login session。
4. `/logout` 清除登录态。

### 操作流程

1. 用户打开 `http://localhost:8085`。
2. 前端调用 `/api/status` 判断登录状态。
3. 未登录时显示登录面板，用户输入用户名和密码。
4. 登录成功后页面刷新并进入主工作台。
5. 点击退出登录时调用 `/logout`，完成后刷新页面。

### 输入输出规则

1. 登录输入：JSON，包含 `username` 和 `password`。
2. 登录成功输出：`{"success": true, "username": "..."}`
3. 登录失败输出：HTTP 401，`{"success": false, "message": "Invalid credentials"}`
4. 状态输出：未登录为 `{"logged_in": false}`，已登录为 `{"logged_in": true, "username": "..."}`

### 边界与异常

1. 后端初始化时如果没有 `admin` 用户，会自动创建默认账号 `admin / 123456`。
2. `SECRET_KEY` 当前写死为 `your-secret-key-here`，生产或局域网共享时必须改为随机密钥。
3. Session cookie 名称可通过环境变量 `KMATRIX_SESSION_COOKIE_NAME` 覆盖，默认是 `kmatrix_8085_session`。

## 4.2 工作台导航与通用交互

### 功能范围

左侧导航按业务分组：

1. 核心数据：知识库管理、机型矩阵管理、多媒体预览。
2. 质量管控：知识库评分、知识库治理。
3. 工具与映射：数据设置、智能映射。
4. 日志与归档：修改记录、归档管理。

### 操作流程

1. 用户登录后进入默认的知识库管理页面。
2. 点击导航项切换模块，前端更新标题、描述和模块内容。
3. 页面中的弹窗用于编辑、配置、导入、提交确认、归档命名、标签筛选等。
4. 通用 Toast 用于显示成功、失败、警告和进度提示。

### 辅助能力

1. 防抖和节流：搜索、筛选、滚动等高频操作通过防抖/节流优化。
2. 表格列宽：多媒体表格列宽可持久化到 localStorage。
3. 分页缓存：知识库分页缓存最多 10 页，缓存 5 分钟。
4. 下载动作：导出类接口通常通过 `window.location.href` 或 fetch blob 下载。
5. 弹窗保护：知识库编辑关闭前会检查未保存状态。

## 4.3 知识库管理

### 功能范围

知识库管理是系统的主数据维护模块，覆盖：

1. V1/V1T-1 表切换。
2. 分页查询、搜索、筛选和排序。
3. 新增、编辑、软删除。
4. 批量选择和批量操作。
5. 标签查看、筛选和维护。
6. 数据导入、查重预览、模板下载和导出。
7. 同步此刻到前刻。
8. 一键同步下游。
9. 完成本轮修订。
10. AI 优化、Markdown 编辑和快捷录入。

### 查询与筛选

支持字段：

1. `table`：`knowledge_base_v1` 或 `knowledge_base_v1_t1`。
2. `page`、`pageSize`：分页参数。
3. `id`：按 `question_wiki_id` 模糊匹配。
4. `question`：按问题模糊匹配。
5. `similar_question`：在相似问数组中做 Python 侧匹配。
6. `answer`：按答案模糊匹配。
7. `product`：按产品型号匹配。
8. `category`：按产品品类匹配，支持多分类。
9. `url`：聚合图片、视频、文件、跳转链接后匹配。
10. `review_status`：按修订状态筛选。
11. `tags`：按知识库标签筛选。
12. `sort_by`、`sort_dir`：排序字段和方向。

输出包括：

1. `data`：知识库记录列表。
2. `total`：总数。
3. `page`、`pageSize`：分页信息。
4. 每条记录会附带 `kb_tags`。

边界条件：

1. 只允许访问 V1 和 V1T-1 两张表。
2. 排序字段有 allowlist，非法字段会回退到默认排序。
3. 相似问搜索会把候选记录拉回后在 Python 侧匹配，数据量较大时性能依赖分页策略。
4. 标签筛选在标签表缺失或查询失败时会返回空结果或错误提示。

### 新增与编辑

主要接口为 `/api/kb/update` 和 `/api/kb/item`。

操作流程：

1. 用户打开编辑弹窗。
2. 填写或修改问题、答案、问题类型、答案类型、BM25、相似问、错误列表、关键词、图片链接、视频链接、文件链接、跳转链接、产品品类和产品型号。
3. 系统校验问题/答案是否为空，校验产品型号是否在型号库中。
4. 已有记录会读取保存前快照，比较变更字段。
5. 无实际变更时返回 `no_change`。
6. 有变更时写入主表，并同步写入 `knowledge_base_modifications` 修改记录。
7. 新增记录生成 `ICWIKIYYYYMMDDnnnn` 格式 ID，状态置为 `creating`。
8. 修改记录状态置为 `modifying`。

输入规则：

1. 列表字段包括 `similar_questions`、`error_list`、`keyword_list`、`image_urls`、`video_urls`、`file_urls`，可输入数组或可解析文本。
2. `if_bm25` 转为布尔值。
3. 产品型号可用英文逗号、中文逗号分隔。
4. 产品型号会按型号库归一化、去空格、去重和排序。
5. `update_time` 在保存时更新为当前时间。

输出规则：

1. 成功返回 `success: true`，并包含 `id` 或更新结果。
2. 无变更返回 `success: true` 且提示无变化。
3. 型号非法返回 HTTP 400，并列出未知型号。
4. 主库客户端不可用返回 HTTP 400 或 500。

异常处理：

1. 修改记录写入失败时，主数据可能已保存，接口会返回 warning 或后台日志。
2. 远端/主库字段不匹配时，写修改记录会尝试丢弃未知列后重试。
3. 保存已有记录但数据库中找不到该 ID 时返回 404。

### 删除

知识库删除是软删除，不会立即物理删除主表记录。

操作流程：

1. 用户选择记录并点击删除。
2. 后端读取原始记录。
3. 写入修改记录，`change_type = delete`。
4. 将主表记录 `review_status` 设置为 `deleting`。

边界条件：

1. 不存在的 ID 返回错误。
2. 删除后记录仍在 V1 中，可通过状态筛选看到。
3. 真正清理需要依赖后续治理或人工流程。

### 修订状态

系统使用 `review_status` 表示知识变更阶段：

1. `unadjusted`：未调整或已完成修订。
2. `creating`：新建待修订确认。
3. `modifying`：修改中。
4. `deleting`：删除中。

“完成本轮修订”会批量把 `creating`、`modifying`、`deleting` 重置为 `unadjusted`。

### 知识库标签

标签相关接口：

1. `GET /api/kb/tags`：获取所有知识库标签名称。
2. `GET /api/kb/item/tags`：获取某个条目的标签。
3. `PUT /api/kb/item/tags`：替换某个条目的标签。

输入规则：

1. 标签名称会 trim。
2. 允许前端传入不存在的标签名称，后端会自动创建。
3. 单条记录标签数量上限为 200。
4. 标签绑定区分 `library_type`：`current` 对应 V1，`previous` 对应 V1T-1。

输出规则：

1. 获取接口返回数组。
2. 更新接口返回更新后的标签列表。

异常处理：

1. ID 或表类型缺失返回 400。
2. 标签表不存在或数据库不可用会返回错误。

### AI 优化与 Markdown 编辑

编辑弹窗中支持：

1. Markdown 工具栏：辅助编辑答案文本。
2. AI 优化问题：优化问题表达。
3. AI 优化答案：按通用处理、故障类、使用类、功能类任务优化答案。
4. AI 优化相似问：生成或整理相似问题。
5. AI 判断问题类型：基于问题类型配置 JSON 输出分类。
6. 快捷录入：从 APP 操作库或产品操作库插入标准步骤。

AI 输出规则：

1. 普通接口返回 JSON。
2. 流式接口返回文本事件流。
3. 答案优化要求输出结构化 JSON，包括答案、URL、备注等字段。
4. 相似问优化输出 `items`。
5. 问题类型可根据 `question_type_config_json` 的分类树返回 `full_class`。

边界条件：

1. 未配置 AI API Key 或 Base URL 时，AI 优化和评分连接测试会失败。
2. 模型返回非 JSON 时，后端会尝试提取 JSON；提取失败返回错误。
3. AI 功能依赖外部网络或本地兼容服务，不属于离线必然可用功能。

## 4.4 知识库数据导入、查重、导出与同步

这是系统最复杂、风险最高的模块之一。

### 4.4.1 导入入口

接口：`POST /api/kb/import`

支持文件：

1. `.xlsx`
2. `.xls`
3. `.csv`

支持模式：

1. `upsert`：增量更新。ID 存在则更新，ID 不存在则新增。
2. `append`：批量新增。ID 已存在则跳过，只插入新记录。
3. `overwrite`：全量覆盖。先执行 V1 到 V1T-1 备份，备份成功后清空 V1 和评分表，再导入新数据。

### 4.4.2 字段映射

导入时支持中文和英文表头，最终落入数据库字段如下：

| 导入表头 | 数据库字段 | 说明 |
| --- | --- | --- |
| ID、问题编号、question_wiki_id | question_wiki_id | 知识库主键；为空时记录不带 ID 导入，具体是否可成功取决于数据库约束 |
| 问题类型、question_type | question_type | 问题分类 |
| 问题、question | question | 用户问题 |
| 答案、answer | answer | 标准答案 |
| 答案类型、answer_type | answer_type | text/image/video 会映射为 文本/图片/视频 |
| BM25、bm25、if_bm25 | if_bm25 | 布尔字段 |
| 相似问题、相似提问、similar_questions | similar_questions | JSONB/list 字段 |
| 错误列表、error_list | error_list | JSONB/list 字段 |
| 关键词、keyword_list | keyword_list | JSONB/list 字段 |
| 图片链接、image_urls | image_urls | JSONB/list 字段 |
| 视频链接、video_urls | video_urls | JSONB/list 字段 |
| 文件链接、file_urls | file_urls | JSONB/list 字段 |
| 跳转链接类型、link_type | link_type | 外链类型 |
| 跳转链接（url/key）、URLs、URL链接、link_url | link_url | 外链 URL 或 key |
| 更新时间、更新时刻、update_time | update_time | 时间字段 |
| 型号、产品型号、产品名称、机型、products、product_name | product_name | 适用型号，多个型号用逗号分隔 |
| 产品品类、产品分类、分类、product_categories、product_category_name | product_category_name | 产品品类 |

特殊规则：

1. 导入时始终忽略 `review_status`，防止 Excel 覆盖系统修订状态。
2. 检测到旧字段 `urls` 或 `answer_info` 时拒绝导入，要求改用图片/视频/文件/跳转链接新字段。
3. 导入前会检查主库是否包含 `image_urls`、`video_urls`、`file_urls`、`link_type`、`link_url` 字段；缺失时提示执行 Schema 更新 SQL。

### 4.4.3 字段清洗规则

1. 产品型号和产品品类：清理逗号后的多余空格，将 `"nan"` 转为空。
2. 答案类型：`text`、`image`、`video` 分别转为 `文本`、`图片`、`视频`。
3. 型号校验：依据型号库归一化匹配，去空格、大小写不敏感；未知型号会阻断导入。
4. 产品品类回填：如果 `product_category_name` 为空，会根据 `product_name` 反查型号库自动回填品类；命中多个品类时用英文逗号连接。
5. JSON/list 字段：既支持 JSON 数组，也支持逗号、中文逗号、换行分隔的普通文本。空字符串、`nan`、`null` 会转为空。
6. BM25：`1/true/yes/是` 转 true，`0/false/no/否` 转 false，空值和未知值默认 false。
7. 时间字段：datetime 会转 ISO 字符串；空时间字段会删除，不写入。
8. 跳转链接：`link_url` 为空时置空；有 `link_url` 但无 `link_type` 时默认 `外部链接`。
9. 批内重复 ID：同一个文件中重复出现的 `question_wiki_id` 只保留最后一条；无 ID 记录保留。

### 4.4.4 导入模式处理

`upsert`：

1. 先按文件中的 ID 查询主库现有 ID。
2. 每 100 条一批 upsert。
3. 统计新增和更新 ID。
4. 被本次触达的 ID 最后统一把 `review_status` 重置为 `unadjusted`。

`append`：

1. 先查询已有 ID。
2. 已存在 ID 跳过并计入 skipped。
3. 不存在 ID 批量 insert。
4. 被导入的新 ID 状态重置为 `unadjusted`。

`overwrite`：

1. 前端先调用 `POST /api/kb/import/preview` 读取导入文件，展示导入记录数、当前 V1 记录数、评分缓存数量、重复 ID、缺失 ID 和未知型号数量。
2. 只有用户在高风险弹窗中确认后，实际导入请求才会携带 `confirm_overwrite=true` 和 `expected_incoming_count`。
3. 后端再次校验导入记录数；若确认时和执行时数量不一致，返回 HTTP 409 要求重新确认。
4. 调用与“同步此刻到前刻”相同的 V1 到 V1T-1 备份逻辑。
5. 备份失败时立即终止，不清空 V1。
6. 清空 `kb_scores` 前，会先把评分缓存写入 `instance/backups/kb_import_overwrite/kb_scores_*.json`。
7. 清空依赖表 `kb_scores`。
8. 尝试全局删除 `knowledge_base_v1`，再循环校验和批量删除残留。
9. 插入新数据。
10. 被导入 ID 状态重置为 `unadjusted`。
11. 如果批量写入失败，系统会尝试用 V1T-1 自动恢复 V1，并用评分备份恢复 `kb_scores`，同时在响应中返回 `restore` 和 `score_backup_path`。

异常输出：

1. 文件缺失：HTTP 400。
2. 主库未配置：HTTP 400。
3. 未知型号：HTTP 400，列出前 10 条异常。
4. 旧字段：HTTP 400。
5. 未确认覆盖：HTTP 409，返回覆盖预览，V1 不会被清空。
6. 覆盖前备份失败：HTTP 500，V1 不会被清空。
7. 评分缓存备份失败：HTTP 500，V1 不会被清空。
8. 批量写入部分失败：HTTP 500，返回 `errors`、`stats`、`details`、`restore` 和评分备份路径。若自动恢复失败，需要根据返回信息和备份文件人工恢复。

1. question

3. question_type

6. similar_questions

9. image_urls
10. video_urls
11. file_urls
12. link_type
13. link_url
14. product_name
15. product_category_name

边界与风险：

1. 查重接口目前只读取 Excel，不支持 CSV 查重。
2. 查重和正式导入已共用同一份字段映射：`型号`、`产品型号`、`产品名称`、`机型` 均映射到 `product_name`，`产品品类`、`产品分类`、`分类` 映射到 `product_category_name`。
3. 查重不会写入数据。

### 4.4.6 模板下载

接口：`GET /api/kb/template`

功能：生成标准导入模板，包含示例数据和列宽。

当前状态：

3. 如果未安装 `xlsxwriter`，自动使用项目已依赖的 `openpyxl` 生成模板，并保留基础列宽设置。

### 4.4.7 导出

接口：`GET /api/kb/export`

功能：导出知识库 CSV。

输入参数：

1. `table`：V1 或 V1T-1。
2. `ids`：指定 ID 列表。
3. `product`、`keyword`、`question`、`answer`、`review_status`：筛选条件。
4. `columns`：指定导出列。

输出规则：

1. 返回 CSV 文件。
2. 列表字段会展开为文本。
3. 结果为空时返回空 CSV。

### 4.4.8 同步此刻到前刻

接口：`POST /api/kb/sync`

功能：把 `knowledge_base_v1` 全量复制到 `knowledge_base_v1_t1`。

流程：

1. 清空 V1T-1。
2. 从 V1 读取允许字段。
3. 分批插入 V1T-1。
4. 同步标签关系：`current` 标签复制为 `previous`。

边界条件：

1. 只同步 allowlist 字段，避免未知字段污染备份表。
2. 主库不可用时失败。
3. 这是覆盖式备份，执行前应确认 V1 是当前希望保留的状态。

### 4.4.9 一键同步下游

接口：`POST /api/kb/sync_downstream`

功能：将当前 V1 同步至下游模块。

同步顺序：

1. 机型矩阵：执行矩阵 `merge` 同步。
2. 多媒体预览：扫描 V1 的图片、视频、文件和跳转链接，更新 `link_previews`。
3. 评分表：同步 `kb_scores` 快照，新增未评分，标记内容变化为 outdated，删除主库不存在的评分记录。
4. 治理提示：月度召回仍需在治理页导入 Excel；本接口只刷新治理关联所需的评分快照。

冲突策略：

1. 矩阵 `merge` 保留手动修改的矩阵项。
2. 多媒体同步按 URL 去重，已有 URL 会更新 KB 关联和系统标签。
3. 评分同步不会立即重跑 LLM，只调整评分任务状态。

异常处理：

1. 任一步失败即终止后续步骤并返回失败。
2. 返回 `steps` 字段，展示各步骤执行结果。

## 4.5 产品型号库与机型分类映射

## 4.6 机型矩阵管理

### 功能范围

机型矩阵把知识库记录和产品型号形成二维表：

1. 行：知识库问题。
2. 列：产品型号。
3. 单元格：该问题是否适用于该型号。

核心功能：

1. 查看矩阵。
2. 搜索和筛选。
3. 单格编辑。
4. 批量设置已配置/未配置。
5. 待提交差异计算。
6. 提交变更。
7. 提交日志和明细。
8. 添加列、复制列、克隆配置。
9. 同步矩阵。
10. 导出矩阵。
11. 产品和分类统计。

### 查询与筛选

接口：`GET /api/matrix/data`

筛选维度：

1. ID、问题、答案搜索。
2. 产品型号搜索。
3. 产品分类 chips。
4. 映射分类。
5. 表头型号多选。
6. 修改状态：手动修改/未修改。
7. 差异比较：找出矩阵状态与 V1 `product_name` 不一致的数据。

表头收缩逻辑：

1. 搜索问题只筛选行，不收缩列。
2. 搜索机型只筛选行，不收缩列。
3. 产品分类会筛选行并收缩表头型号列。
4. 映射分类会筛选行，并要求记录关联型号集合与映射集合完全一致。
5. 表头型号多选只收缩列，不筛选行。
6. 多个列白名单条件叠加时取交集。

### 单格编辑

接口：`POST /api/matrix/update`

输入：

1. `question_wiki_id`
2. `product_name`
3. `is_configured`

流程：

1. 拉取最新矩阵和日志缓存。
2. 查找该问题和型号的矩阵行。
3. 如果不存在，尝试用同问题的其他矩阵行复制问题、答案、分类和更新时间作为模板。
4. 更新 `is_configured`。
5. 设置 `manual_edit = true`、`edit_source = cell`。
6. 写入本地 SQLite。
7. 如果启用矩阵主库同步，后台 upsert 到 `matrix_column` 和 `product_matrix`。

输出：

1. 成功返回 `{"success": true}`。
2. 缺少 ID 或产品返回 400。

异常与风险：

1. 后台同步失败只打印日志，前端已收到成功，可能产生本地和主库不一致。
2. 如果找不到同问题模板，新建行的内容字段可能为空。

### 批量编辑

接口：`POST /api/matrix/batch_update`

输入：

1. `question_wiki_ids`：问题 ID 数组。
2. `product_name`：目标型号。
3. `is_configured`：目标状态。

流程：

1. 批量查询已有矩阵行。
2. 已有行更新状态，设置 `manual_edit = true`。
3. 如果原来源不是单格编辑，则设置 `edit_source = bulk`。
4. 缺失行从同问题其他行复制内容后创建。
5. 启用主库同步时后台 upsert。

输出：

1. 成功返回更新数量。
2. 缺少参数返回 400。

### 待提交差异

接口：`POST /api/matrix/mismatch_changes`

功能：计算手动编辑后的矩阵值与 V1 `product_name` 实际适用型号之间的差异。

输入：

1. 可选 `wiki_ids`。不传时扫描所有手动编辑行。

输出字段：

1. `question_wiki_id`
2. `product_name`
3. `old_is_configured`：V1 当前是否包含该型号。
4. `new_is_configured`：矩阵当前单元格状态。
5. `edit_source`：`cell` 或 `bulk`

边界条件：

1. 只输出当前值与 V1 不一致的差异。
2. 非手动编辑且无有效来源的行不参与提交。
3. 如果矩阵列集合存在，非列内型号会被忽略。

### 提交变更

接口：`POST /api/matrix/submit_changes`

输入：

1. `operation_id`：可选；不传自动生成 UUID。
2. `attempt`：尝试次数。
3. `changes`：差异数组。

单条差异必填字段：

1. `question_wiki_id`
2. `product_name`
3. `old_is_configured`：布尔值。
4. `new_is_configured`：布尔值。
5. `edit_source`

校验规则：

1. `changes` 必须是非空数组。
2. ID 和型号不能为空。
3. old/new 必须是布尔值。
4. old 和 new 不能相同。
5. 同一 `(question_wiki_id, product_name)` 不能重复。
6. 提交前会再次读取当前矩阵值，如果当前库值与待提交 after 不一致，则判为冲突。

写入流程：

1. 创建或更新 `MatrixSubmitOperation`，状态为 pending。
2. 写入 `Button` 明细表，记录 before/after、来源、提交人和时间。
3. 如果启用 button 同步，写入主库 `button` 表。
4. 操作成功后把提交操作状态改为 success。
5. 如果启用矩阵主库同步，同步 `matrix_submit_operation` 和 `button`。
6. 根据变更构造 `knowledge_base_modifications`，来源为 `机型矩阵管理`。

输出：

1. 成功返回 `operation_id`、写入数量和 Supabase/主库同步信息。
2. 校验失败返回 HTTP 400，包含 errors。
3. 提交失败返回 HTTP 500，并把操作状态记录为 failed。

重要说明：

矩阵提交会写入修改记录，但不会直接改写 `knowledge_base_v1.product_name`。后续需要通过修改记录审核/合并导出或人工流程完成主库落地。

### 提交日志与明细

接口：

1. `GET /api/matrix/submit_logs`
2. `GET /api/matrix/submit_logs/<operation_id>/details`

功能：

1. 查看每次提交的 operation_id、状态、重试次数、操作人、错误信息和时间。
2. 查看提交明细，包括问题、答案、型号、变更前后、来源、提交人和快照差异。

边界条件：

1. 日志默认最多返回 50 条，可通过 limit 调整到 1-200。
2. 明细默认最多 500 条，最大 2000 条。

### 添加列、复制列与克隆配置

添加列：`POST /api/matrix/add_column`

1. 输入 `product_name`。
2. 若列已存在返回 400。
3. 新列 sort_order 为当前最大值 + 1。

复制列：`POST /api/matrix/copy_column`

1. 输入 `source_product` 和 `target_product`。
2. 如果目标列不存在，自动创建。
3. 把源型号中已配置的问题复制为目标型号已配置。
4. 已存在但未配置的目标单元格会改为已配置。
5. 来源标记为 bulk。

克隆配置：`POST /api/matrix/clone_config`

1. 支持 `mode = model`：从单个源型号复制。
2. 支持 `mode = category`：从产品分类或映射分类复制。
3. `strategy = append`：只补充目标缺失或未配置项。
4. `strategy = force_sync`：强制同步目标，并删除源集合中不应保留的配置。
5. 返回更新数量、删除数量和 pending_changes。

边界风险：

1. `force_sync` 会删除部分源配置，使用前必须确认目标范围。
2. 克隆结果仍需提交，才会进入提交日志和修改记录闭环。

### 同步矩阵

接口：`POST /api/matrix/sync`

模式：

1. `merge`：增量合并，保留手动编辑；新增 V1 适用型号会补入矩阵；V1 中消失且非手动编辑的矩阵行会删除。
2. `reset`：清空矩阵，完全按 V1 `product_name` 重新生成；手动配置会丢失。
3. `content_refresh`：刷新问题、答案、品类、更新时间，同时新增/删除条目；会保留已有配置状态。

输出：

1. `added`
2. `updated`
3. `deleted`

异常处理：

1. 主库未连接返回失败。
2. 同步过程中异常会 rollback SQLite 事务。

### 导出矩阵

接口：`GET /api/matrix/export`

功能：导出矩阵 Excel。

当前状态：

1. `/api/matrix/export` 已只保留一个 Flask 路由：`export_matrix_data`。
2. 导出会跟随当前矩阵页面筛选条件，包括 ID、问题、答案、机型多选、匹配模式、产品品类、映射分类、列筛选、修改/未修改筛选和差异比较。
3. Excel 生成优先使用 `xlsxwriter`，未安装时自动降级使用 `openpyxl`。

### 产品、分类和统计

接口：

1. `GET /api/matrix/products`：返回型号库、矩阵列和矩阵数据中的型号并集。
2. `GET /api/matrix/categories`：返回型号库和矩阵数据中的品类并集。
3. `GET /api/matrix/stats`：按型号或分类统计配置数量。

统计规则：

1. 型号统计按 `product_name` 计数，并对部分空格差异做 fallback。
2. 映射分类统计要求问题关联型号集合与映射型号集合完全一致。
3. 普通品类统计按 `product_category` 去重问题 ID。

当前链接更新/删除已返回真实写入结果。剩余风险主要来自主库连接中断或权限配置错误，前端会显示失败提示。

### 从知识库同步链接

接口：`POST /api/links/sync_kb`

扫描字段：

1. `image_urls`
2. `video_urls`
3. `file_urls`
4. `link_url`

同步规则：

1. 对 URL 去空格、去反引号并去重。
2. 同一 URL 可关联多个 KB ID，`kb_id` 用逗号拼接。
3. 根据来源加系统标签：`来源:图片链接`、`来源:视频链接`、`来源:文件链接`、`来源:外部链接`。
4. 跳转链接会根据 `link_type` 加 `外链类型:*` 标签。
5. 根据后缀或 URL 特征识别 image、video、youtube、file、link。
6. 新 URL 插入，已有 URL 更新。
7. 已不存在于知识库的 URL 不删除，只清空 KB 关联。

### 图片代理

接口：`GET /api/proxy_image?url=...`

功能：后端代请求外部 URL，以解决跨域和资源预览问题。

当前风险：

1. 没有登录保护。
2. SSRF 检查只写了注释和 `pass`，没有真正阻止 localhost、127.0.0.1、192.*、10.*。
3. 请求使用 `verify=False`，禁用 TLS 验证。
4. 支持 Range 透传，适合视频拖动，但也扩大了代理能力。

### 链接导入

接口：`POST /api/links/import`

当前实现已统一写入 PostgreSQL `link_previews`。导入规则为：读取 JSON 数组，逐条清理 URL；缺失 `id` 时自动生成 UUID；缺失 `type` 时按 URL 后缀和 YouTube 特征识别 image、video、youtube、file、link；`tags` 支持数组、JSON 字符串或逗号分隔字符串；通过 `upsert(..., on_conflict='id')` 写入主列表使用的同一张表。

异常处理：

1. 请求体不是数组：HTTP 400。
2. 主库未配置：HTTP 500。
3. 空 URL 或非法条目会计入 `skipped`，不阻断其他有效链接。
4. 数据库写入失败时返回 HTTP 500，不再写入旧 SQLite `Link` 表，也不会返回“假成功”。

4. 只在评分成功时保存缓存。

2. 用户在高风险弹窗中确认后，`POST /api/scoring/clear_cache` 必须携带 `confirm_clear=true` 和 `expected_count`。
3. 后端再次校验当前数量；若与确认时数量不一致，返回 HTTP 409 要求重新确认。
4. 删除前会把完整 `kb_scores` 写入 `instance/backups/scoring_cache/kb_scores_*.json`。
5. 返回值包含 `deleted` 和 `backup_path`。

## 5.2 导入冲突解决策略

1. 文件内重复 ID：保留最后一条。
2. 主库存在同 ID：
   - upsert：覆盖更新该 ID 的导入字段。
   - append：跳过。
   - overwrite：先清空后插入。
3. 型号冲突：导入前校验，未知型号直接阻断，不写入。
4. 字段格式冲突：JSON/list 字段尽量解析和清洗；不可解析时按分隔文本拆分。
5. 旧字段冲突：`urls`、`answer_info` 直接阻断。
6. Schema 冲突：缺少新链接字段时阻断。
7. 覆盖导入失败：可能产生部分批次写入成功，需要根据返回 errors 和 stats 做人工核查。

## 5.3 同步触发条件

1. 手动点击“同步此刻到前刻”：触发 V1 到 V1T-1 全量覆盖。
2. 全量覆盖导入前：自动触发 V1 到 V1T-1 备份。
3. 手动点击“一键同步下游”：触发矩阵、多媒体和评分同步。
4. 手动点击“同步机型矩阵”：按 merge/reset/content_refresh 同步矩阵。
5. 手动点击“同步数据”或评分同步：同步 `kb_scores`。
6. 手动点击“同步知识库链接”：同步 `link_previews`。
7. 保存型号库：同步矩阵列并清理不再存在的型号数据。
8. 矩阵单格/批量编辑：本地保存后可后台同步矩阵表。

## 5.4 备份机制

1. V1T-1：知识库前刻备份，适合覆盖导入前和发布前保留当前状态。
2. 归档批次：保存修改记录历史，适合审计和阶段性清空当前待处理记录。
3. SQLite 系统库：保存登录、矩阵本地数据、提交日志、归档 fallback 等。应纳入文件级备份。
4. 配置 JSON：保存型号库、映射、AI Key 和评分 Prompt，应纳入备份但注意敏感信息。
5. PostgreSQL 主库：应通过 pg_dump 或数据库管理工具做定期备份。系统内没有自动数据库备份计划。

## 7. 常见问题排查 FAQ

### Q1：为什么打开页面后一直显示登录？

检查：

1. 后端是否在 8085 端口启动。
2. 浏览器是否禁用了 cookie。
3. `/api/status` 是否返回 401 或非 JSON。
4. `SECRET_KEY` 是否频繁变化导致 session 失效。

### Q2：默认账号是什么？

如果 SQLite 用户表中没有 admin，系统初始化会创建 `admin / 123456`。建议首次部署后立即修改或禁用默认密码机制。

### Q3：为什么知识库列表加载失败？

检查：

1. `supabase_config_local.json` 的本地 PostgreSQL 配置是否正确。
2. PostgreSQL 是否启动。
3. `knowledge_base_v1` 表是否存在。
4. 是否执行了最新 Schema，特别是链接拆分字段和 `review_status`。

### Q4：模板下载依赖什么？

当前 `/api/kb/template` 会优先使用 `xlsxwriter`，如果环境未安装该库，会自动降级使用 `openpyxl`。正常情况下不再需要额外安装 `xlsxwriter` 即可下载模板。

### Q5：为什么查重预览和正式导入结果不一致？

查重接口对 `型号` 的字段映射与导入接口不一致。查重把 `型号` 当作产品品类，导入把 `型号` 当作适用型号。修复前建议模板明确使用 `产品名称` 表示型号，使用 `产品品类` 表示品类。

### Q6：为什么矩阵改了以后知识库 product_name 没变？

矩阵提交只写提交日志和修改记录，不直接改写 V1。需要通过修改记录审核、智能合并导出或后续人工流程把变更落到主库。

### Q7：为什么多媒体链接删除后页面显示成功，但数据库没变化？

部分链接更新/删除使用后台线程，接口会先返回成功。后台失败只写日志。建议操作后刷新列表确认。

### Q8：为什么 AI 评分或优化失败？

检查：

1. API Key 是否配置。
2. Base URL 是否是 OpenAI 兼容接口。
3. 模型名是否正确。
4. 本机网络是否允许访问该服务。
5. 模型是否按要求返回 JSON。

### Q9：归档后修改记录不见了，是否丢失？

归档会把当前修改记录迁移到归档表，并从当前修改记录中删除。可在归档管理中按批次查看和导出。

### Q10：为什么对象存储上传/下载不可用？

本地 PostgreSQL 模式已提供文件系统 fallback：`storage_upload` 写入 `instance/storage/<bucket>/<object_path>`，`storage_download` 从同一路径读取，并进行 bucket 名称和路径穿越校验。它不是 Supabase Storage 的权限系统，只适合本地版文件读写兼容。

### Q11：为什么沙箱内测试 PostgreSQL 连接失败，但实际应用可用？

本次检查中，普通沙箱命令连接 localhost:5432 失败，解除沙箱网络限制后连接成功。这是 Codex 沙箱隔离导致，不代表 PostgreSQL 服务不可用。

## 9. 问题、风险与隐患清单

### P0/P1 高优先级

1. 默认账号和硬编码密钥风险  
   系统会自动创建 `admin / 123456`，且 Flask `SECRET_KEY` 写死为 `your-secret-key-here`。局域网共享或生产使用时存在会话伪造、默认密码登录风险。

2. `/api/proxy_image` 存在 SSRF 和 TLS 风险  
   该接口未登录保护，localhost/内网地址检查没有真正阻断，并且 `requests.get(..., verify=False)` 禁用证书校验。应立即加登录保护、URL allowlist/denylist、内网 IP 解析拦截和 TLS 验证。

### P2 中优先级

3. 多媒体链接导入写入遗留 SQLite 表：已修复  
   `/api/links/import` 已统一写入 `link_previews`，并返回真实数据库写入结果。

4. 后台写入返回过早：已修复主要写入入口  
   链接更新/删除、矩阵单格/批量更新和矩阵列新增不再使用后台线程静默写入；远端同步失败会通过响应 `warnings` 返回，前端会提示。

5. 本地 PostgreSQL 过滤语法不完整：已增强  
   `_parse_filter` 已支持复杂 `or/and`、JSONB `cs.`、`->>`、`::text`，并增加 SQL 标识符统一转义。仍应避免把未审计的任意用户字段名直接透传为过滤字段。

6. 本地 storage 不支持：已修复  
   本地模式 upload/download 已落到 `instance/storage`，包含 bucket 和路径穿越校验。

7. 全量覆盖导入风险高：已加保护  
   overwrite 已增加导入预览、强制确认、V1T-1 备份、评分缓存 JSON 备份和失败自动恢复尝试。

8. 型号库保存会删除矩阵数据：已加保护  
   保存前增加差异预览和二次确认；后端未收到确认时拒绝破坏性清理。

9. 清空评分缓存风险高：已加保护  
   清空前必须读取数量、二次确认并生成本地 JSON 备份；后端校验 `expected_count`。

10. 归档是破坏性迁移：已加保护  
   归档前展示待归档记录数和删除影响；后端强制 `confirm_archive` 和 `expected_count`。

### P3 低优先级或改进项

11. 单角色权限模型不足  
   当前登录后可访问全部业务模块，不适合多人协作和敏感操作隔离。

12. AI/评分/embedding 依赖外部服务  
   离线环境下相关功能不可用，应在 UI 上显示配置状态和降级提示。

13. 配置和密钥明文保存  
   数据库密码、API Key 和 Prompt 配置都在本地 JSON 中明文保存，应至少限制文件权限，长期可改为环境变量或密钥管理。

14. 写入类接口缺少统一审计  
   大多数核心写入有修改记录，但配置类、操作库、部分链接类操作审计不足。

15. 第二轮更广接口抽测受审批服务影响未完成  
   本次尝试继续抽测更多只读/无副作用接口时，桌面审批服务返回 502，未继续重复请求。已完成的一轮抽测覆盖了主要页面入口。
